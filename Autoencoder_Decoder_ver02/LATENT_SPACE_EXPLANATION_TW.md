# Latent Space 維度說明（繁體中文）

## 實際的 Latent 維度

**模型架構流程：**

```
輸入圖片：[B, T, 1, 128, 128] = 每張圖片 16,384 個像素
         ↓
FrameEncoder（編碼器）：
  - 第1層 Conv: 128×128 → 64×64（空間縮小一半）
  - 第2層 Conv: 64×64 → 32×32（空間再縮小一半）
  - 第3層 Conv: 32×32 → 16×16（空間再縮小一半）
         ↓
中間表示：[B, 128, 16, 16]
  - 128 個通道（feature maps）
  - 16×16 的空間解析度
  - 總共：128 × 16 × 16 = 32,768 個數值
         ↓
AdaptiveAvgPool2d(1): 16×16 → 1×1（空間池化）
  - 把每個 16×16 的區域平均成 1 個數值
  - 結果：[B, 128, 1, 1] = 128 個數值
         ↓
Linear 投影: 128 → 256
         ↓
最終 Latent：[B, 256] = 256 維向量
```

## 「256×16×16」是什麼意思？

**這是一個誤解！** 實際上：

1. **中間表示（在池化之前）：**
   - 形狀：`[B, 128, 16, 16]`
   - 128 個通道 × 16×16 空間 = 32,768 個數值
   - **但這不是最終的 latent！**

2. **最終 Latent（在池化之後）：**
   - 形狀：`[B, 256]`
   - 256 維向量
   - **這才是真正的 latent representation**

## 為什麼會有「256×16×16」的誤解？

可能的原因：

1. **看錯了中間層的輸出：**
   - 中間層確實有 16×16 的空間解析度
   - 但這在 `AdaptiveAvgPool2d(1)` 後被池化掉了

2. **混淆了不同版本的模型：**
   - 可能有其他版本的模型使用空間 latent（保留 16×16）
   - 但目前的模型使用向量 latent（256 維）

3. **看錯了文檔：**
   - 某些文檔可能描述的是中間表示，不是最終 latent

## 實際的 Latent 維度

**每張圖片：**
- 輸入：128×128 = **16,384 像素**
- Latent：**256 維向量**
- 壓縮比：16,384 / 256 = **64× 壓縮**

**整個序列（16 張圖片）：**
- Latent sequence：`[B, 16, 256]`
- 總共：16 × 256 = **4,096 個數值**

## 為什麼用 AdaptiveAvgPool2d？

**目的：** 把空間資訊壓縮成單一向量

**好處：**
1. **降低維度：** 從 32,768 → 256（128× 壓縮）
2. **空間不變性：** 不管胚胎在圖片中的位置，latent 都一樣
3. **計算效率：** 向量運算比空間張量運算快
4. **適合 LSTM：** LSTM 需要向量輸入，不是空間張量

**如果不用池化：**
- Latent 會是 `[B, 128, 16, 16]` = 32,768 維
- 這會讓模型更容易學習 identity function
- 而且 LSTM 無法直接處理空間張量

## 總結

- **最終 Latent：** 256 維向量（每張圖片）
- **中間表示：** 128×16×16（在池化之前，不是最終 latent）
- **壓縮比：** 64×（16,384 → 256）
- **為什麼不是 identity：** 因為 256 維無法儲存所有 16,384 像素的資訊
